# Módulo: Core (Configuración, Autenticación, Infraestructura Transversal)

**Propósito:** Piezas transversales usadas por todas las rutas — settings/secretos (`config.py`), autenticación/autorización JWT+Authentik (`auth_utils.py`), rate limiting (`rate_limit.py`), Vault (`vault.py`), logging, base de datos.
**Dependencias Clave:** `pydantic-settings`, `PyJWT`, `hvac` (Vault), `PostgresUserRepository`.

---

## Bitácora de Cambios Significativos (Changelog Local)

_Formato:_ `- **[YYYY-MM-DD]** ([Nombre del desarrollador]): [Descripción breve del cambio estructural]`

- **[2026-07-22]** (Claude / auditoría de seguridad): **Eliminado un backdoor de escalación de privilegios** en `auth_utils.get_current_user` — cualquier cuenta cuyo email/username contuviera las substrings `"israelm"`, `"victor"` o `"millan"` recibía rol `admin` automáticamente en cada login, incluso revirtiendo revocaciones hechas desde el panel "Gestión de Usuarios". El rol ahora se asigna exclusivamente vía `POST /admin/users/{id}/role`. El equivalente en frontend (`authStore.js`) también fue eliminado — ver README de `frontend/src/stores/`.
- **[2026-07-22]** (Claude / auditoría de seguridad): Se agregó la dependencia centralizada `require_admin(current_user = Depends(get_current_user))` para que ninguna ruta vuelva a reimplementar el chequeo de rol admin de forma inconsistente.
- **[2026-07-22]** (Claude / auditoría de seguridad): `get_current_user` ahora normaliza `roles` a una lista real (`_normalize_roles`) antes de devolverlo, en vez de dejar que cada archivo consumidor parseara el JSON serializado por su cuenta (fuente de inconsistencias).
- **[2026-07-22]** (Claude / auditoría de seguridad): `config.py` — si Vault no responde o no incluye `SECRET_KEY`/`JWT_SECRET`/`DB_PASSWORD` **en un entorno `APP_ENV=production`**, el proceso ahora aborta el arranque en vez de continuar silenciosamente con los fallbacks hardcodeados (predecibles y presentes en el repo). En desarrollo (`APP_ENV=development`, el default de `docker-compose.yml`) el comportamiento no cambia.
- **[2026-07-22]** (Claude / auditoría de seguridad): credenciales del widget movidas de literales en `routes/auth.py` a `settings.WIDGET_EMAIL`/`settings.WIDGET_PASSWORD` (override vía Vault soportado). **Pendiente operativo:** rotar la contraseña real en la BD, ya que el valor anterior quedó expuesto en el historial del repositorio.
- **[2026-07-22]** (Claude / auditoría de seguridad): nuevo módulo `rate_limit.py` — limitador de tasa en memoria (sin dependencia nueva de pip), aplicado a los endpoints que invocan LLM o procesan archivos. Limitación conocida y documentada en el propio archivo: el contador es por-proceso, así que con >1 worker el límite efectivo se multiplica por el número de workers; migrar a Redis si el despliegue real usa múltiples workers/réplicas.
- **[2026-07-22]** (Claude / auditoría de seguridad, no aplicado): la verificación TLS al validar el JWKS de Authentik sigue deshabilitada (`ssl.CERT_NONE` en `auth_utils.py`) — se documentó el hallazgo y la razón para no des-habilitarlo a ciegas (falta el certificado real de la CA interna en este entorno para verificar contra él); ver comentario en el propio archivo.
- **[2026-07-22 tarde]** (Claude, revisión de funcionalidades de Google Antigravity): `database.py::init_db()` solo ejecutaba `Base.metadata.create_all` (crea tablas nuevas, nunca altera una existente). Se verificó por introspección directa contra la BD real que `chat_messages.feedback_rating/feedback_text` (columnas nuevas del feedback 👍/👎) no existían en la tabla real, aunque el modelo ORM sí las declaraba — cualquier request a esos campos fallaba con error real de PostgreSQL. Se agregó `_apply_schema_patches()` (ALTER TABLE ADD COLUMN IF NOT EXISTS, idempotente) ejecutado en cada arranque junto a `create_all`, y se aplicó el parche directamente contra la BD real en este momento. Este mecanismo es una mitigación, no reemplaza migraciones versionadas — recomendado inicializar Alembic (ya está en `requirements.txt` pero nunca se configuró) en la próxima iteración mayor de esquema.
- **[2026-07-22 tarde]** (Claude, revisión): nuevo módulo `crypto.py` (Fernet, clave derivada de `SECRET_KEY`) — `ChatbotProfileModel.custom_api_key` (API key de LLM personalizada por tema, usada activamente en `chat_service.py` para llamadas reales) se guardaba y usaba en texto plano; ahora se cifra al guardar y se descifra solo en memoria para enmascarar/usar. Compatible hacia atrás (si un valor no descifra como token Fernet válido, se usa tal cual). No había ningún tema con la clave configurada en la BD real, por lo que no fue necesaria una migración de datos.
